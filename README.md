# po-suite

Product Owner skill plugin for Claude Code. Provides guided feature discovery, epic decomposition, backlog refinement, and scope checking — all grounded in the **Small Batch & Fast Feedback** principle.

## What It Does

po-suite helps you break work into the smallest manageable size that produces value. Every feature, epic, or idea gets decomposed into issues that each produce a **human-reviewable PR targeting ~500 lines changed**.

This is the product ownership layer that sits upstream of implementation workflows like [Spec-Driven Development (SDD)](https://github.com/liatrio-labs/spec-driven-workflow). It ensures work enters implementation already right-sized and well-defined.

## Installation

Add to your project's `skills.txt`:

```
PCritchfield/po-suite
```

Or install directly:

```bash
npx skills add PCritchfield/po-suite
```

## Skills

| Command | Purpose | When to use |
|---------|---------|-------------|
| `/po-discover` | Guided feature discovery → structured feature brief | Starting from a vague idea or client request |
| `/po-breakdown` | Decompose feature/epic → right-sized issues | Have a feature brief or epic, need implementable issues |
| `/po-refine` | Periodic backlog health check | Reviewing existing issues for staleness, sizing, readiness |
| `/po-scope-check` | Interactive scope check for task lists | After generating tasks, before starting implementation |

## The PO Agent

Each skill embeds a Product Owner agent persona that provides product-focused judgment:

- Evaluates business value, priority, and right-sizing
- Asks product questions (not technical ones)
- Grounds assessments in the Small Batch standard
- Never makes technical design decisions, writes specs, or implements code

The agent definition lives in `agents/po-agent.md`. Skills embed relevant subsets of this persona for their specific workflows.

## Workflow

```
 Vague idea
     │
     ▼
 /po-discover  ──→  Structured Feature Brief
     │
     ▼
 /po-breakdown ──→  Right-sized issues (~500 line PRs)
     │                        │
     ▼                        ▼
 For each issue:         Suggests: /SDD-1-generate-spec
     │
     ▼
 /SDD-1-generate-spec  ──→  Spec
     │
     ▼
 /SDD-2-generate-task-list  ──→  Task list
     │
     ▼
 /po-scope-check  ──→  Interactive scope review
     │
     ▼
 /SDD-3-manage-tasks  ──→  Implementation
     │
     ▼
 /po-refine (periodic)  ──→  Backlog health check
```

## Artifact Conventions

po-suite uses a `.po/` directory at the project root:

```
<project-root>/.po/
├── config.md          # Active sizing scale, engagement metadata
├── briefs/            # po-discover output (feature briefs)
└── issues/            # po-breakdown output (issue artifacts)
```

- **`config.md`** — Created on first run of any `po-*` skill. Stores the active sizing scale.
- **`briefs/`** — Feature briefs from `/po-discover`, named `<feature-name>-brief.md`.
- **`issues/`** — Issue artifacts from `/po-breakdown`, named `<issue-title>.md`.

## Configuration

### Sizing Scales

The first time any `po-*` skill runs, it asks which sizing scale to use:

| Scale | Values | When to use |
|-------|--------|-------------|
| T-shirt | XS, S, M, L, XL | Default. Quick relative sizing. |
| Fibonacci | 1, 2, 3, 5, 8, 13 | Teams that use story points. |
| Custom | User-defined | You define your own values. |

The choice is saved to `.po/config.md` and used by all skills for that engagement.

## SDD Integration

po-suite integrates with the [Spec-Driven Development workflow](https://github.com/liatrio-labs/spec-driven-workflow) at two points:

1. **`/po-breakdown`** suggests `/SDD-1-generate-spec` as the next step for each issue.
2. **`/po-scope-check`** reviews task lists generated by `/SDD-2-generate-task-list-from-spec` before implementation begins.

### Automatic Scope Check

To have `/po-scope-check` suggested automatically after SDD-2, add this line to your project's `CLAUDE.md`:

```
After /SDD-2-generate-task-list-from-spec completes, invoke po-scope-check
before proceeding to /SDD-3-manage-tasks.
```

Without this line, `/po-scope-check` is available as a manual command but won't be suggested automatically.

## Standalone Usage

po-suite works without the SDD workflow, without the Watch Council, and without any specific persona configuration. It is a standalone plugin that can be installed in any Claude Code environment.

- `/po-discover` and `/po-breakdown` work entirely independently.
- `/po-refine` works on any set of markdown issue files.
- `/po-scope-check` can review any task list, not just SDD-generated ones.

## The Small Batch Standard

All skills enforce the same standard: **an issue should produce a PR that is human-reviewable in one sitting, targeting ~500 lines changed** (additions + modifications + deletions per `git diff --stat`).

This is a conversational guideline, not a hard gate. It's rooted in Liatrio's Small Batch & Fast Feedback principle.

See `shared/small-batch.md` for the full definition.
